# Phase 1: Data Model Design

**Feature**: Smart Contract LLM Builder for Flow/Cadence
**Date**: 2025-09-25

## Core Entities

### User
Represents system users with different persona types and access levels.

**Attributes**:
- `user_id` (UUID): Primary identifier
- `email` (string): User contact information
- `persona_type` (enum): EXPERT | INTERMEDIATE | BEGINNER | NON_TECHNICAL
- `flow_account_address` (string, optional): User's Flow blockchain address
- `created_at` (timestamp): Account creation time
- `preferences` (json): User interface and behavior preferences
- `data_retention_consent` (boolean): Consent for data usage in learning

**Relationships**:
- One-to-Many with ContractSubmission
- One-to-Many with DeploymentLog
- One-to-One with UserDataControl

### ContractSubmission
Represents a user's contract input for processing.

**Attributes**:
- `submission_id` (UUID): Primary identifier
- `user_id` (UUID, foreign key): Submitting user
- `input_type` (enum): CDC_FILE | SOL_FILE | NATURAL_LANGUAGE | MIXED
- `content` (text): Contract content or natural language description
- `pre_conditions` (json, optional): User-defined preconditions
- `post_conditions` (json, optional): User-defined postconditions
- `status` (enum): PENDING | PROCESSING | COMPLETED | FAILED
- `created_at` (timestamp): Submission time
- `processed_at` (timestamp, optional): Processing completion time

**Relationships**:
- Many-to-One with User
- One-to-Many with DeploymentLog
- One-to-One with GeneratedConfiguration

### DeploymentLog
Structured record of contract deployment attempts and results.

**Attributes**:
- `log_id` (UUID): Primary identifier
- `submission_id` (UUID, foreign key): Associated contract submission
- `deployment_id` (string): Flow deployment identifier
- `network` (string): Flow network (testnet/mainnet/emulator)
- `status` (enum): SUCCESS | FAILED | TIMEOUT | VALIDATION_ERROR
- `error_message` (text, optional): Error details if failed
- `error_code` (string, optional): Standardized error code
- `transaction_hash` (string, optional): Blockchain transaction hash
- `gas_used` (integer, optional): Gas consumption for deployment
- `execution_time_ms` (integer): Deployment execution time
- `log_content` (text): Full deployment log output
- `created_at` (timestamp): Deployment attempt time

**Relationships**:
- Many-to-One with ContractSubmission
- Many-to-One with GeneratedConfiguration

### DocumentationKnowledgeBase
Indexed Flow blockchain and Cadence documentation.

**Attributes**:
- `doc_id` (UUID): Primary identifier
- `source` (string): Documentation source URL
- `title` (string): Document title
- `content_type` (enum): LANGUAGE_SPEC | API_REFERENCE | TUTORIAL | EXAMPLE
- `content` (text): Document content
- `embedding_vector` (array): Vector representation for semantic search
- `chunk_index` (integer): Chunk index for large documents
- `last_updated` (timestamp): Last update time
- `version` (string): Documentation version

**Relationships**:
- Standalone entity with no direct relationships

### GeneratedConfiguration
flow.json compatible configuration files generated by LLM.

**Attributes**:
- `config_id` (UUID): Primary identifier
- `submission_id` (UUID, foreign key): Source contract submission
- `config_content` (json): Complete flow.json configuration
- `generated_contract_code` (text): Generated .cdc contract code
- `validation_status` (enum): VALID | INVALID | PENDING
- `validation_errors` (json, optional): Validation error details
- `created_at` (timestamp): Generation time
- `last_modified` (timestamp): Last modification time

**Relationships**:
- One-to-One with ContractSubmission
- One-to-Many with DeploymentLog
- One-to-Many with TransactionProposal

### TransactionProposal
System-generated transaction suggestions for user approval.

**Attributes**:
- `proposal_id` (UUID): Primary identifier
- `config_id` (UUID, foreign key): Source configuration
- `transaction_type` (enum): DEPLOY | UPDATE | INTERACT
- `transaction_data` (json): Complete transaction details
- `estimated_gas` (integer): Estimated gas cost
- `user_approval_status` (enum): PENDING | APPROVED | REJECTED
- `signed_transaction` (text, optional): User-signed transaction
- `created_at` (timestamp): Proposal creation time
- `responded_at` (timestamp, optional): User response time

**Relationships**:
- Many-to-One with GeneratedConfiguration

### LearningFeedbackLoop
System that analyzes deployment logs to improve contract generation.

**Attributes**:
- `feedback_id` (UUID): Primary identifier
- `submission_id` (UUID, foreign key): Associated contract submission
- `log_id` (UUID, foreign key): Analyzed deployment log
- `pattern_type` (enum): ERROR_PATTERN | SUCCESS_PATTERN | OPTIMIZATION_OPPORTUNITY
- `insights` (json): Extracted insights and recommendations
- `confidence_score` (float): AI confidence in the insight (0.0-1.0)
- `applied_to_generation` (boolean): Whether insight was used in generation
- `created_at` (timestamp): Feedback creation time

**Relationships**:
- Many-to-One with ContractSubmission
- Many-to-One with DeploymentLog

### UserDataControl
User preferences and controls for data retention and deletion.

**Attributes**:
- `control_id` (UUID): Primary identifier
- `user_id` (UUID, foreign key): Associated user
- `data_retention_preference` (enum): KEEP_ALL | DELETE_AFTER_30D | DELETE_IMMEDIATELY
- `learning_consent` (boolean): Consent for anonymized learning
- `deletion_requests` (json): History of data deletion requests
- `last_data_access` (timestamp, optional): Last time user data was accessed
- `created_at` (timestamp): Control setup time
- `updated_at` (timestamp): Last preference update time

**Relationships**:
- One-to-One with User

## Data Validation Rules

### ContractSubmission Validation
- `input_type` must be valid enum value
- `content` must be non-empty and within size limits (10MB max)
- `pre_conditions` and `post_conditions` must be valid JSON if provided
- `status` transitions must follow workflow rules

### DeploymentLog Validation
- `status` must have corresponding `error_message` if FAILED
- `transaction_hash` must be valid Flow transaction hash format if provided
- `execution_time_ms` must be positive integer
- `log_content` must be structured JSON for automated parsing

### GeneratedConfiguration Validation
- `config_content` must be valid flow.json schema
- `generated_contract_code` must pass Cadence syntax validation
- `validation_errors` must be array if validation fails

### User Data Validation
- `email` must be valid email format
- `persona_type` must match user's actual skill level
- `flow_account_address` must be valid Flow address format if provided

## State Transitions

### ContractSubmission Lifecycle
```
PENDING → PROCESSING → COMPLETED
                ↓
               FAILED
```

### TransactionProposal Lifecycle
```
PENDING → APPROVED → EXECUTED
         ↓
        REJECTED
```

### UserDataControl States
```
ACTIVE → RETENTION_UPDATED → DATA_DELETED
```

## Indexing Strategy

### Performance-Critical Indexes
1. **User**: `user_id` (primary), `email` (unique)
2. **ContractSubmission**: `user_id`, `status`, `created_at`
3. **DeploymentLog**: `submission_id`, `status`, `network`, `created_at`
4. **DocumentationKnowledgeBase**: `embedding_vector` (vector index), `content_type`
5. **GeneratedConfiguration**: `submission_id`, `validation_status`

### Search Optimization
- Full-text search on documentation content
- Vector similarity search for semantic queries
- Composite indexes for common query patterns
- Time-based partitioning for deployment logs

## Security & Privacy Considerations

### Data Encryption
- Sensitive user data encrypted at rest
- API keys and credentials stored in secure vaults
- Transaction proposals signed client-side only

### Access Control
- Role-based access control (RBAC) for different user personas
- Data segregation between users
- Audit logging for all data access

### Compliance
- GDPR-compliant data deletion mechanisms
- User consent management for learning data
- Data anonymization for model training

## Scalability Considerations

### Horizontal Scaling
- Database sharding by user_id for user data
- Vector database scaling for documentation storage
- Queue-based processing for deployment operations

### Performance Optimization
- Connection pooling for database access
- Caching layer for frequently accessed documentation
- Async processing for long-running operations
